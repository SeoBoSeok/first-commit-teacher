# 선생님 노하우 — 웹 서비스 & 클로드코드 실전 영역

> 학생B는 이미 기능을 빠르게 붙이는 힘이 있다. 선생님 버전이 더할 것은 **"안 되는 경우"를 다루는 습관**과
> **운영 가능한 서비스로 가는 마지막 20%**다.

## 1. 이미 잘하고 있는 것 (유지·강화)

- ✅ `.env`·`data/` git 제외, 비밀번호 bcrypt 해시 — **보안 기본기가 습관**이 돼 있다
- ✅ `store.ts`의 주석: "Prisma 임시 대체, 호출부 무변경 전제" — **임시 코드에 의도를 남기는 것**은
  시니어들도 잘 못하는 일급 습관. 이 주석 덕분에 M1 전환 설계가 이미 끝나 있다
- ✅ 세계관을 코드(데이터 구조)로 정리 — IP 프로젝트의 정석

## 2. 웹 서비스의 노하우

### 돈이 오가는 코드의 3원칙 (Stripe)
1. **서명 검증**: 웹훅은 누구나 POST할 수 있다. `stripe.webhooks.constructEvent`로 서명 확인 없이는 한 줄도 처리하지 않는다
2. **멱등성**: 같은 이벤트가 두 번 와도 주문은 한 건 — 이벤트 ID를 저장해 중복을 거른다
3. **금액은 서버가 결정**: 클라이언트가 보낸 가격을 믿지 않는다. 상품 ID만 받고 가격은 서버 데이터에서

### 인증 토큰의 수명 관리
- 인증 메일 토큰: 만료 시간 + 1회용 (사용 즉시 무효화) + 재발송 시 이전 토큰 폐기
- 실패 화면을 먼저 디자인한다 — "만료됐어요, 다시 보내드릴까요?"가 500 에러보다 백배 낫다

### 거대 컴포넌트 분리 (SpacekkabbiApp.tsx 934줄)
- 한 번에 다 쪼개지 않는다 — **건드리는 기능이 있을 때 그 부분만** 컴포넌트로 추출 (보이스카우트 규칙)
- 분리 순서 추천: 데이터는 이미 분리됨(잘함) → 섹션 단위(Hero/Factions/Characters/Shop) → 상태는 마지막에
- "use client"는 잎(leaf)으로 밀어낸다 — 서버 컴포넌트가 기본값

### 인터페이스 고정 전환 (M1의 핵심 기술)
```
store.ts의 함수 시그니처(getUser, createOrder…)는 그대로 두고
내부 구현만 fs → Prisma로 교체 → 호출하는 쪽은 한 줄도 안 바뀐다
```
이게 되면 "구현은 갈아끼울 수 있다"는 감각이 생긴다 — 아키텍처 감각의 시작.

## 3. 클로드코드 사용 노하우 (이 저장소의 작업 규칙)

- 기능 시작 전 `docs/01-PRD.md` 대조 — 범위 밖이면 만들지 않는다
- **돈·인증 코드는 Plan Mode로**: 결제·인증을 건드릴 땐 계획을 먼저 검토받는다
- 프롬프트에 실패 케이스를 같이 준다: "결제 성공 처리해줘" ❌ →
  "웹훅에서 서명 검증 후 주문 생성, 같은 이벤트 중복 수신 시 무시하게 해줘" ✅
- Next.js 16은 학습 데이터보다 새 버전 — **`node_modules/next/dist/docs/` 동봉 문서를 먼저 읽게** 한다 (AGENTS.md)
- 환경변수 추가 시: `.env.example`에 키 이름만 기록 (값은 절대 커밋 금지)

## 4. 흔한 함정

| 함정 | 처방 |
|------|------|
| localhost에서만 영원히 개발 | M3(배포)를 뒤로 미루지 않는다 — 배포가 버그를 드러낸다 |
| 해피 패스만 테스트 | 테스트 카드 4000…0002(거절)로 실패 경로를 먼저 |
| AI가 준 인증 코드를 이해 없이 머지 | 돈·인증 코드는 "이 코드 설명해줘" 의무화 |
| 비밀키를 코드에 하드코딩 | 환경변수 + .env.example 패턴 고수 (지금까지 잘하고 있음) |
| 기능 추가 욕심 | 까비 월드와 역할 분담 기억 — 커뮤니티 기능은 월드 영역 |
